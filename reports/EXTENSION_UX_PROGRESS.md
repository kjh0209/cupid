## Summary

Full Cursor/Antigravity-grade UX redesign of the `vscode-extension/` delivered in v0.3.0.

## Before / After

### Before (v0.2.0 — root cause bugs)
- Apply button created a new `Untitled-1` document instead of modifying the active file
- LLM returned partial 2-line snippets; apply had no intent inference
- UI: plain text, no markdown rendering, no syntax highlighting
- No stop button; status bar was static
- No inline edit capability
- System prompts allowed partial snippets (no format enforcement)

### After (v0.3.0 — this PR)
- **Apply** correctly detects `path=` / `mode=` attributes and routes to the right file
- **SEARCH/REPLACE patch** mode with fuzzy matching; confirmation dialog before any write
- **Structured Edit Protocol** enforced in system prompts for 5 task types
- **Inline Diff Decoration** with accept/reject CodeLens at document level
- **Ctrl+K Inline Edit** — quick input → ghost text preview → accept/reject
- **Chat UI**: full markdown rendering, syntax highlighting, code block headers with path/mode, per-block Copy/Apply/Diff buttons, streaming dots, @ file mention, attachment chips, Stop button
- **Status bar state machine**: routing→streaming→idle/error with animated spinner
- **Onboarding**: connection dot, example prompt chips, backend status in empty state

## Known VS Code API Limitations

1. **Inline InputBox positioning**: VS Code does not support positioning an `InputBox` inline at the cursor location. Ctrl+K shows the prompt at the top of the editor window (same as VS Code's native behavior). Ghost text still appears at the cursor line.

2. **Per-hunk accept/reject**: Implemented as document-level "Accept all / Reject all" via CodeLens. True per-hunk granularity would require a full diff library (not included to avoid adding dependencies). Can be added in v0.4.0.

3. **@ mention file content**: File content is read by the extension and injected into the prompt automatically when attachments are present. Large files are truncated at 8000 chars.
