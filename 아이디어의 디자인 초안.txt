CUPID의 디자인은 “AI를 많이 쓰게 만드는 IDE”가 아니라 “AI를 가장 합리적으로 쓰게 만드는 IDE”라는 관점에서 출발한다. 따라서 UI/UX의 중심은 단순한 chat window가 아니라, 개발자가 AI 사용의 비용, 품질, 문맥을 한눈에 이해하고 통제할 수 있는 decision-aware interface이다. CUPID는 기존 개발자가 익숙한 VS Code 기반 workflow를 유지하되, 그 위에 AI model routing layer, cost dashboard, context memory panel, model decision log, code diff verification panel을 결합한 형태로 설계된다.

기본 화면은 세 영역으로 구성된다. 첫째, 중앙에는 기존 IDE와 동일하게 code editor가 위치한다. 개발자는 기존처럼 파일을 열고 코드를 수정하며, AI가 생성한 diff를 직접 확인할 수 있다. 둘째, 우측에는 CUPID Copilot panel이 배치된다. 이 panel에서 사용자는 자연어 prompt를 입력하고, CUPID는 해당 요청을 분석하여 선택된 모델, 선택 이유, 예상 비용, 예상 latency, risk level을 표시한다. 예를 들어 “fix the authentication middleware”라는 요청이 들어오면 CUPID는 이를 security-sensitive backend bug fix로 분류하고, 단순 UI 수정과 달리 mid/strong tier 모델을 선택한 이유를 보여준다. 셋째, 상단 또는 side panel에는 Cost & Context Dashboard가 배치된다. 여기에는 이번 요청에서 절감된 비용, baseline model 대비 절감률, input/output token 수, 사용된 context snippet, repo memory hit 여부, 이전 decision log 반영 여부가 표시된다.

CUPID의 핵심 UX는 “Send” 버튼을 누른 뒤 보이지 않는 곳에서 자동으로 일어나는 네 단계로 설계된다. 1단계에서 Prompt Optimizer가 사용자의 요청을 압축하고 명확한 작업 지시로 바꾼다. 예를 들어 “혹시 이거 좀 봐줄 수 있어? 로그인 쪽이 이상한 것 같은데 최대한 전체 구조 안 건드리고 고쳐줘”라는 요청은 “Fix login-related authentication bug. Keep diff minimal. Do not modify unrelated architecture. Use existing auth middleware pattern.”과 같은 구조화된 prompt로 변환된다. 2단계에서 Task Classifier가 작업 유형, 난이도, 위험도, 필요한 context 범위를 판단한다. 3단계에서 Context Preservation Layer가 현재 파일, 관련 dependency, 과거 수정 이력, architecture decision, coding convention 중 필요한 정보만 추출한다. 4단계에서 Model Router가 비용·품질·속도 점수를 계산해 최적 모델을 선택하고 실행한다.

CUPID의 디자인에서 중요한 차별점은 모델 선택 결과를 숨기지 않는다는 점이다. 많은 AI coding tool은 사용자가 내부적으로 어떤 모델이 호출되었는지, token이 얼마나 쓰였는지, 왜 비용이 발생했는지 알기 어렵다. 반면 CUPID는 “이번 작업은 단일 파일 UI 수정이므로 Gemini/Haiku 계열의 cost-efficient model을 선택했습니다”, “이번 작업은 DB schema migration과 관련되어 있어 strong model floor가 적용되었습니다”, “history summarization으로 8,200 token을 절감했습니다”와 같이 모델 선택의 이유를 자연어로 설명한다. 이는 단순 기능이 아니라 신뢰 형성 장치이다. 개발자는 CUPID가 무작정 싼 모델을 쓰는 것이 아니라, 실패 비용과 작업 복잡도를 고려해 합리적 선택을 하고 있음을 이해할 수 있다.

또한 CUPID는 memory의 시각화를 중요하게 설계한다. 기존 AI chat은 대화가 길어질수록 어떤 맥락이 모델에 전달되는지 사용자가 알기 어렵다. CUPID는 Context Storage panel을 통해 현재 작업에 포함된 repo summary, relevant files, previous decisions, accepted changes, rejected alternatives를 보여준다. 사용자는 필요 없는 context를 제외하거나, 특정 파일을 강제로 포함하거나, “이 프로젝트에서는 Prisma schema를 직접 수정하지 말고 migration을 통해 반영한다”와 같은 팀 규칙을 memory에 저장할 수 있다. 이 기능은 개인 개발자에게는 반복 설명을 줄여주고, 팀 단위 개발에서는 코드 스타일과 아키텍처 일관성을 유지하는 데 기여한다.

CUPID의 시각적 정체성은 “개발자의 낭비되는 AI 비용을 빨아들이는 black hole”이라는 브랜드 이미지로 설계된다. 로고의 보라색 원형 그래픽은 여러 모델로 흩어지는 AI 사용을 하나의 intelligent routing layer로 모아주는 portal을 상징한다. 슬로건 “Don’t code stupid. Use Cupid.”은 개발자를 비판하는 문장이 아니라, 비효율적 모델 사용을 직관적으로 지적하는 강한 메시지다. 다만 실제 사업계획서와 공식 문서에서는 이를 “Don’t overpay for intelligence you don’t need” 또는 “Use the right AI for every coding task”처럼 정제된 메시지로 병행 사용할 수 있다.

최종적으로 CUPID의 디자인은 다음 사용자 경험을 목표로 한다. 개발자는 기존 IDE를 떠나지 않는다. 모델을 직접 고르지 않는다. 같은 설명을 반복하지 않는다. AI 비용을 추측하지 않는다. 대신 CUPID가 작업을 이해하고, 필요한 문맥을 가져오고, 적절한 모델을 고르고, 비용과 품질을 검증하며, 그 모든 과정을 투명하게 보여준다. 이것이 CUPID가 단순 AI coding assistant가 아니라 AI-native development operating layer로 설계되는 이유이다.
